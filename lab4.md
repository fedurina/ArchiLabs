# Лабораторная работа №4
## Тема: Проектирование REST API
**Цель:** Получить опыт проектирования программного интерфейса.

Проект: *Система управления бронированиями и номерным фондом для high-end курортов и спа-отелей*.

---

## 1) Принятые проектные решения (не менее 8)

Ниже перечислены решения, принятые при проектировании API. Они применимы к реализации `/api/v1/...`.

1. **Версионирование API в URL:** `/api/v1/...`  
   **Причина:** безопасная эволюция контрактов и возможность поддерживать обратную совместимость.

2. **REST-ориентация на ресурсы (существительные), а не “действия”:** `rooms`, `bookings`  
   **Причина:** понятная модель CRUD и предсказуемые URL.

3. **Единый формат ошибок (Problem Details-стиль):**
   - поля: `type`, `title`, `status`, `detail`, `instance`, `traceId`, опционально `errors[]`
   **Причина:** одинаковая обработка ошибок в клиентах и логах.

4. **Использование стандартных HTTP-кодов:**
   - `200 OK`, `201 Created`, `204 No Content`, `400 Bad Request`, `401 Unauthorized`, `404 Not Found`, `409 Conflict`
   **Причина:** клиенты могут опираться на стандартную семантику протокола.

5. **Аутентификация через заголовок `Authorization: Bearer <token>` (упрощённо для лабы):**  
   **Причина:** отражает реальный подход (JWT/OAuth2), при этом не усложняет лабораторную.

6. **Идемпотентность создания бронирования через `Idempotency-Key` (опционально):**  
   Повторный `POST /bookings` с тем же ключом вернёт уже созданный ресурс.  
   **Причина:** защита от повторов/таймаутов сети (актуально для бронирований).

7. **Фильтрация/пагинация коллекций через query-параметры:**
   - `limit`, `offset` + фильтры (например, `status`, `room_type`, `floor`)  
   **Причина:** управляемый размер ответа и поддержка типовых сценариев поиска.

8. **PUT для полного обновления ресурса, DELETE для удаления ресурса:**  
   - `PUT /rooms/{id}`, `DELETE /rooms/{id}`  
   - `PUT /bookings/{id}`, `DELETE /bookings/{id}`  
   **Причина:** соответствие семантике HTTP-методов, прозрачная CRUD-модель.

9. **Бизнес-валидации на уровне API:**
   - проверка `check_in_date < check_out_date`  
   - проверка доступности номера по датам (конфликт → `409`)  
   - запрет удаления номера с активными бронированиями (конфликт → `409`)  
   **Причина:** корректность данных и предсказуемые ошибки.

10. **Трассировка запросов через `X-Trace-Id` и поле `traceId` в ошибках:**  
    **Причина:** упрощает поиск проблем в логах (особенно при интеграциях).

## 2) Документация по API

### 2.1. Общие соглашения
- Base URL: `http://127.0.0.1:8000`
- Версия: `/api/v1`
- Формат данных: `application/json`
- Авторизация (для всех запросов):  
  `Authorization: Bearer demo-token`

### 2.2. Ресурс Rooms (Номера)

#### 2.2.1. GET /api/v1/rooms
**Описание:** Получить список номеров с фильтрами и пагинацией.  
**Query-параметры (все необязательные):**
- `status`: `available|reserved|occupied|needs_cleaning|cleaning_in_progress|ready|out_of_service`
- `room_type`: `standard|comfort|lux`
- `floor`: integer
- `limit`: integer (по умолчанию 50)
- `offset`: integer (по умолчанию 0)

**Ответ 200:**
```json
[
  {
    "id": "uuid",
    "number": "305",
    "floor": 3,
    "room_type": "lux",
    "status": "available"
  }
]
```

**Ошибки:** `401`

---

#### 2.2.2. POST /api/v1/rooms
**Описание:** Создать номер.  
**Body:**
```json
{
  "number": "305",
  "floor": 3,
  "room_type": "lux",
  "status": "available"
}
```

**Ответ 201:** объект `RoomOut` (как выше).

**Ошибки:**  
- `401` — нет авторизации  
- `409` — номер с таким `number` уже существует


---

#### 2.2.3. GET /api/v1/rooms/{room_id}
**Описание:** Получить номер по `room_id`.  
**Ответ 200:** объект `RoomOut`.  
**Ошибки:** `401`, `404`.

---

#### 2.2.4. PUT /api/v1/rooms/{room_id}
**Описание:** Полностью обновить данные номера.  
**Body:** как в POST.  
**Ответ 200:** обновлённый `RoomOut`.  
**Ошибки:** `401`, `404`, `409`.

---

#### 2.2.5. DELETE /api/v1/rooms/{room_id}
**Описание:** Удалить номер.  
**Ответ 204:** без тела.  
**Ошибки:** `401`, `404`, `409` (если есть активные бронирования).

---

### 2.3. Ресурс Bookings (Бронирования)

#### 2.3.1. GET /api/v1/bookings
**Описание:** Получить список бронирований с фильтрами и пагинацией.  
**Query-параметры (необязательные):**
- `guest_id`: integer
- `room_id`: uuid
- `status_`: `created|confirmed|cancelled`
- `date_from`: `YYYY-MM-DD`
- `date_to`: `YYYY-MM-DD`
- `limit`, `offset`

**Ответ 200:**
```json
[
  {
    "id": "uuid",
    "guest_id": 42,
    "room_id": "uuid",
    "check_in_date": "2026-04-01",
    "check_out_date": "2026-04-05",
    "status": "created"
  }
]
```
---

#### 2.3.2. POST /api/v1/bookings
**Описание:** Создать бронирование на конкретный номер.  
**Headers (опционально):** `Idempotency-Key: <string>`  
**Body:**
```json
{
  "guest_id": 42,
  "room_id": "uuid",
  "check_in_date": "2026-04-01",
  "check_out_date": "2026-04-05"
}
```

**Ответ 201:** объект `BookingOut`.  
**Ошибки:**  
- `401`  
- `404` — номер не найден  
- `409` — номер недоступен на даты или `out_of_service`  
- `400` — некорректный диапазон дат


---

#### 2.3.3. GET /api/v1/bookings/{booking_id}
**Описание:** Получить бронирование по `booking_id`.  
**Ответ 200:** объект `BookingOut`.  
**Ошибки:** `401`, `404`.

---

#### 2.3.4. PUT /api/v1/bookings/{booking_id}
**Описание:** Полностью обновить бронирование (в рамках полей модели).  
**Body:**
```json
{
  "guest_id": 42,
  "room_id": "uuid",
  "check_in_date": "2026-04-02",
  "check_out_date": "2026-04-06",
  "status": "confirmed"
}
```
**Ответ 200:** обновлённый `BookingOut`.  
**Ошибки:** `401`, `404`, `409`, `400`.


---

#### 2.3.5. DELETE /api/v1/bookings/{booking_id}
**Описание:** Удалить бронирование.  
**Ответ 204:** без тела.  
**Ошибки:** `401`, `404`.
