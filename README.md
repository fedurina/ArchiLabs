# Стейкхолдеры

1. **Гости отеля (клиенты высококлассных курортов)**
   Используют систему для поиска номеров, просмотра фотографий и планировки конкретных комнат, бронирования нужного типа номера или конкретной комнаты, а также для управления своим бронированием и открытия номера через мобильный телефон.

2. **Сотрудники ресепшена / службы приёма и размещения**
   Работают с бронированиями, созданными по телефону и лично; помогают гостям при заселении/выезде, уточняют статусы номеров, решают конфликтные ситуации по размещению.

3. **Сотрудники службы уборки (горничные, супервайзеры клининга)**
   Используют систему и специальные устройства на уборочных тележках для получения списка задач по уборке и подготовке номеров с приоритизацией, отмечают выполнение уборки и состояние номера.

4. **Сотрудники технического обслуживания (инженеры, ремонтники)**
   Получают задания по устранению неисправностей в номерах и технических помещениях, обновляют статус задач и состояние комнаты (например, “на ремонте”, “готово”).

5. **Менеджмент отеля (директор, менеджер по размещению, менеджер по эксплуатации)**
   Анализируют загрузку номерного фонда, контролируют качество уборки и технического обслуживания, следят за своевременной подготовкой номеров к заселению, принимают решения на основе отчетов.

6. **Компания по бронированию (владелец системы)**
   Инвестирует в разработку и внедрение системы, определяет бизнес-требования, политику работы с партнёрскими отелями, стратегию развития продукта для премиум-рынка.

7. **ИТ-отдел компании по бронированию (разработчики, DevOps, поддержка)**
   Отвечают за разработку, развертывание, поддержку и сопровождение системы, интеграцию с существующей системой бронирования и смарт-замками, мониторинг и устранение инцидентов.

8. **Операторы колл-центра компании по бронированию**
   Принимают телефонные звонки клиентов, создают и изменяют бронирования в системе от имени гостей.

9. **Поставщик системы умных замков**
   Предоставляет API или протокол интеграции для выдачи и отзыва мобильных ключей, отвечает за корректное открытие дверей номеров по цифровым ключам.

## 2. Перечень функциональных требований

### 2.1. Требования, связанные с гостями

**FR-01. Регистрация и аутентификация гостя**
Система должна позволять гостю создать учётную запись и входить в неё:

* через веб-интерфейс (браузер),
* через мобильное приложение или мобильный веб,
* через обращение по телефону (учётка создаётся оператором),
* при личном обращении на ресепшен (учётка создаётся сотрудником).

**FR-02. Поиск и просмотр доступных номеров**
Система должна предоставлять гостю возможность:

* указать даты заезда и выезда, количество гостей и дополнительные параметры,
* получить список доступных типов номеров и/или конкретных комнат,
* видеть краткую информацию по каждому варианту (цена, площадь, тип кровати, вид из окна и т.п.).

**FR-03. Просмотр детальной информации о конкретном номере**
Система должна позволять гостю:

* просматривать фотографии конкретного номера (реальные фото этой комнаты),
* видеть расположение номера на схеме отеля (этаж, близость к лифту, к спа-зоне и др.),
* видеть список удобств, доступных именно в этом номере.

**FR-04. Бронирование типа номера**
Система должна позволять гостю бронировать не конкретный номер, а тип номера с учётом доступности и цен.

**FR-05. Бронирование конкретного номера**
Система должна позволять гостю выбрать и забронировать конкретный номер при условии, что он доступен на выбранные даты.

**FR-06. Управление бронированием гостем**
Система должна позволять гостю:

* просматривать историю и текущие бронирования,
* изменять бронирование (при наличии такой политики) – даты, тип номера, дополнительные услуги,
* отменять бронирование (по правилам тарифа),
* получать уведомления о подтверждении/изменении/отмене.

**FR-07. Выдача мобильного ключа для смарт-замка**
При успешном заселении система должна инициировать выдачу цифрового ключа на мобильный телефон гостя для открытия дверного замка его номера, а при выезде – отзывать этот ключ.

---

### 2.2. Требования, связанные с персоналом отеля

**FR-08. Создание и управление бронированиями персоналом**
Система должна позволять сотрудникам ресепшена и операторам колл-центра:

* создавать бронирования от имени гостей (по телефону и при личном визите),
* изменять и отменять бронирования,
* видеть все детали бронирования (в т.ч. выбранный конкретный номер).

**FR-09. Управление статусами номеров**
Система должна поддерживать статусы номера, например:

* “Доступен”
* “Забронирован”
* “Занят”
* “Требуется уборка”
* “Уборка в процессе”
* “Готов к заселению”
* “На ремонте/вне эксплуатации”

Сотрудники ресепшена, уборки и технического обслуживания должны иметь возможность просматривать и изменять статусы в рамках своей роли.

**FR-10. Просмотр планируемой загрузки номерного фонда**
Система должна предоставлять персоналу и менеджменту:

* календарный вид занятости номеров,
* информацию, когда номер будет нужен в следующий раз (дата следующего заезда, и т.п.),
* отчёты по загрузке (по типам номеров, по датам, по сезону).

---

### 2.3. Управление уборкой и обслуживанием

**FR-11. Формирование задач на уборку и обслуживание**
Система должна автоматически формировать задачи для уборки и технического обслуживания номеров на основе:

* событий выезда/заезда,
* текущего статуса номера,
* зарегистрированных дефектов/поломок,
* приоритета (номер нужен через 2 часа / номер нужен завтра).

**FR-12. Распределение задач между сотрудниками**
Система должна распределять задачи между сотрудниками уборки и технического обслуживания с учётом:

* приоритетов по времени (когда номер снова будет нужен),
* географии (расположение комнат по этажам/корпусам),
* текущей загрузки каждого сотрудника.

Распределённые задания должны отображаться на запатентованных устройствах, прикреплённых к тележкам.

**FR-13. Обновление статуса задач и номеров персоналом уборки/обслуживания**
Сотрудники, используя свои устройства, должны иметь возможность:

* просматривать список назначенных задач,
* отмечать начало и завершение задачи,
* менять статус номера (например, “Уборка в процессе” → “Готов к заселению”),
* фиксировать проблемы и инициировать задачу для технической службы.

---

### 2.4. Интеграционные и системные функции

**FR-14. Интеграция с существующей системой бронирования**
Все стандартные функции бронирования (платежи, хранение регистрационных данных, биллинг) должны реализовываться через существующую систему бронирования. Новая система должна:

* передавать данные о создании/изменении/отмене бронирований в эту систему,
* получать информацию о состоянии платежей и статусе бронирования из неё.

**FR-15. Интеграция с системой умных замков**
Система должна взаимодействовать с поставщиком смарт-замков для:

* выдачи цифровых ключей при заселении,
* ограничения доступа по времени (ключ действителен только в период проживания),
* отзыва ключей при выезде или при досрочном завершении бронирования.

**FR-16. Управление доступом и ролями пользователей**
Система должна поддерживать разграничение прав доступа (гости, ресепшен, уборка, техслужба, менеджмент, администраторы системы) и обеспечивать, чтобы каждый тип пользователя видел и мог изменять только то, что ему положено.

## Перечень сделанных предположений 

1. **Существующая система бронирования имеет рабочий API.**
   Есть документированный интерфейс (REST / SOAP или похожий), через который можно создавать, изменять и отменять брони и получать информацию об оплатах.

2. **Телефонные и “личные” бронирования вносятся вручную.**
   Оператор колл-центра или сотрудник ресепшена просто заполняет те же формы в веб-интерфейсе, без отдельной голосовой/IVR-системы.

3. **У гостей есть современные смартфоны с интернетом.**
   Для целевого сегмента (high-end) предполагается, что гости могут пользоваться мобильным приложением и мобильными ключами для смарт-замков.

4. **На старте поддерживается один отель (или небольшое число отелей).**
   Расширение на сеть курортов планируется позже, сейчас фокус на одном высококлассном объекте.

5. **Одновременная нагрузка — сотни гостей и до ~20 сотрудников.**
   Это не экстремально нагруженная система, можно сделать упор на удобство и стабильность работы при такой нагрузке.

6. **Интеграции со смарт-замками и существующей системой бронирования достаточно быстрые и стабильные.**
   Считаем, что внешние API не вносят заметных задержек и не мешают пользовательскому опыту.

7. **Пиковый сезон длится несколько месяцев.**
   Важно успеть с первой версией к ближайшему сезону; дополнительные фичи можно дорабатывать в низкий сезон.

8. **Правила отмены, изменения и оплаты задаёт старая система бронирования.**
   Новая система их не придумывает, а только показывает пользователю и следует им.

9. **Роли персонала задаются крупно, без тонкой настройки.**
   На первом этапе достаточно разделения на группы (ресепшен, уборка, техслужба, менеджмент и т.п.), детальные права по каждому действию не вводятся.

## 5. Перечень нефункциональных требований 

**NFR-01. Производительность и отзывчивость**
Основные действия гостя (поиск номеров, открытие карточки номера, переход к бронированию) должны выполняться примерно за 1–2 секунды при обычной нагрузке.
Изменения статусов номеров и задач для персонала должны отображаться в интерфейсе за несколько секунд, чтобы люди могли оперативно реагировать.

**NFR-02. Доступность и надёжность**
В пиковый сезон система должна быть доступна не менее 99,5 % времени.
При кратких сбоях внешних сервисов (система бронирования, умные замки) система должна не “падать”, а пробовать повторить запросы и корректно восстанавливаться.

**NFR-03. Безопасность и конфиденциальность данных**
Работа идёт только по защищённому протоколу (HTTPS), используется аутентификация и авторизация по ролям.
Персональные данные гостей и данные о бронированиях хранятся с учётом требований вроде GDPR.
Цифровые ключи для смарт-замков шифруются при передаче и хранении, чтобы исключить несанкционированный доступ к номерам.

**NFR-04. Удобство использования (UX) и внешний вид**
Интерфейс для гостей должен выглядеть современно и “премиально”, быть понятен без инструкции и не заставлять долго искать нужные действия.
Интерфейс для персонала — максимально простой и быстрый: крупные кнопки, минимум лишних переходов, нормальная работа на устройствах, закреплённых на тележках уборки.

**NFR-05. Масштабируемость и расширяемость**
Архитектура должна позволять без серьёзных переделок:

* подключать новые отели и дополнительные сервисы (спа, ресторан, трансфер и т.п.),
* заменить текущую систему бронирования или поставщика смарт-замков на другого при сохранении общей логики работы.
